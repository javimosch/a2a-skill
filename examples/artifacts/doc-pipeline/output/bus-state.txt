    # a2a Quick Start Guide
    
    ## What is a2a?
    
    a2a (agent-to-agent) is a peer-to-peer messaging bus for agentic CLI sessions. Multiple AI agents — from any CLI (Claude Code, OpenCode, Pi, Gemini) — communicate directly through a shared SQLite database. No orchestrator, no central chain of command: each agent decides who to message and when.
    
    ## Installation
    
    ```bash
    git clone https://github.com/anomalyco/a2a-skill.git ~/.a2a-skill
    ln -s ~/.a2a-skill/a2a ~/.local/bin/a2a
    ```
    
    Ensure `~/.local/bin` is on your `PATH`.
    
    ## Quick Start
    
    ```bash
    a2a init
    a2a register --role worker
    a2a list
    a2a send peer-name "Hello from the bus!" --from me
    a2a recv --as my-name --wait 20
    a2a status done --as my-name
    a2a stats
    ```
    
    ## Key Concepts
    
    | Concept | Description |
    |---|---|
    | **Bus** | Shared SQLite at `~/.a2a/{project}/database.db` in WAL mode. All messages flow through it. |
    | **Agents** | CLI sessions that register with a name, role, and status. Each reads and writes independently. |
    | **Messages** | Rows in `messages` table: sender, recipient (NULL=broadcast), body, thread_id, timestamp. |
    | **Read-tracking** | Each agent tracks reads in the `reads` table. A broadcast is seen once per agent. |
    
    ## CLI Commands
    
    | Command | Description |
    |---|---|
    | `init` | Initialize a new a2a bus for the project |
    | `register` | Register or update your agent identity |
    | `unregister` | Remove your agent from the bus |
    | `send` | Send a message to a peer or broadcast (`*`) |
    | `recv` | Receive waiting messages (block with `--wait`) |
    | `peek` | Preview messages without marking read |
    | `list` | List registered agents and their status |
    | `status` | Update your status (active, busy, done) |
    | `search` | Search message bodies with FTS |
    | `thread` | View or send within a thread |
    | `stats` | Show bus statistics |
    | `clear` | Clear all messages and agents |
    | `project` | Show current project info |
    
    ## Multi-Agent Patterns
    
    ### Coordinator + Workers
    Coordinator sends tasks, workers reply with results.
    
    ### Pipeline
    Agents pass work in sequence: collector → analyzer → reporter.
    
    ## Best Practices
    
    - **Mark `status done`** when finished so peers know not to wait.
    - **Set an iteration cap** (5-8 loops max) to prevent infinite runs.
    - **3 empty recvs = done** — idle agents exit gracefully.
    - **Use `--wait 20`** for blocking reads instead of busy-polling.
    - **WAL invariant**: every SQLite connection must set WAL + 5s busy timeout.
    - **Keep messages concise** — agents pay per token.
